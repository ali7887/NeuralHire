پروژه از دو سیستم موازی استفاده می‌کند:

NextAuth (src/lib/auth.ts)
فقط برای session plumbing و ساختار پایه

استراتژی: session.strategy = "jwt"
در authorize() یک user object هاردکد شده برمی‌گرداند:
typescript
     { id: "1", email: credentials.email, role: "admin" }
role از طریق callback های jwt و session به توکن و session اضافه می‌شود
این بخش در عمل استفاده نمی‌شود و صرفاً برای سازگاری با NextAuth وجود دارد
Custom JWT System (src/lib/jwt/jwt.utils.ts + src/lib/services/auth.service.ts)
این سیستم اصلی احراز هویت است

دو نوع توکن اصلی دارد: Access Token و Refresh Token
🔑 ساخت و Verify توکن‌ها
Access Token
📍 محل ساخت: src/lib/jwt/jwt.utils.ts → signAccessToken()

typescript
payload: {
  userId: string,
  email: string,
  role: string,
  type: "access"
}

- **مدت اعتبار:** 15 دقیقه
- **Secret:** `JWT_SECRET`
- **Verify:** `verifyAccessToken(token)` در همان فایل
- **الزامات verify:**
  - `userId`, `email`, `role` باید موجود باشند
  - `type === "access"`

#### **Refresh Token**
📍 **محل ساخت:** `src/lib/jwt/jwt.utils.ts` → `signRefreshToken()`

typescript
payload: {
  tokenId: string,  // UUID تصادفی
  userId: string,
  type: "refresh"
}

- **مدت اعتبار:** 30 روز
- **Secret:** `JWT_REFRESH_SECRET`
- **Verify:** `verifyRefreshToken(token)`
- **الزامات verify:**
  - `tokenId`, `userId` باید موجود باشند
  - `type === "refresh"`
- **ذخیره در دیتابیس:** رکورد refresh token در جدول `refreshTokens` ذخیره می‌شود با فیلدهای:
  - `token: tokenId` (UUID)
  - `expiresAt: now + 30 days`
  - `revoked: false`

#### **توکن‌های کمکی**
- **Reset Password Token:** `type: "reset"`, مدت 15 دقیقه, `JWT_RESET_SECRET`
- **Email Verification Token:** `type: "email-verify"`, مدت 30 دقیقه, `JWT_EMAIL_VERIFY_SECRET`

### 🔄 **Flow کامل Login/Register/Logout**

#### **Register** (`src/lib/services/auth.service.ts`)
typescript
register({ email, password, name })

1. بررسی وجود کاربر با همان ایمیل
2. هش کردن رمز عبور با **SHA-256**
3. ایجاد کاربر جدید با `role: "user"`
4. ساخت `refreshId = crypto.randomUUID()`
5. ذخیره refresh token در دیتابیس
6. برگرداندن:
```typescript
   {
user,
accessToken,  // 15m
refreshToken  // 30d
   }
   
Login (src/lib/services/auth.service.ts)
typescript

login({ email, password })

یافتن کاربر با ایمیل
بررسی صحت رمز عبور (مقایسه hash)
ساخت refreshId جدید
ذخیره refresh token در دیتابیس
برگرداندن { user, accessToken, refreshToken }
Logout (src/lib/services/auth.service.ts)
typescript

logout(refreshToken: string)

verify کردن refresh token با verifyRefreshToken()
یافتن رکورد مربوطه در دیتابیس
تنظیم revoked: true برای آن رکورد
توکن دیگر قابل استفاده نیست

👥 Role System (RBAC)
نقش‌های موجود:

"user" — کاربر عادی (پیش‌فرض در register)
"employer" — کارفرما
"admin" — مدیر سیستم
محل تعریف:

در register(): کاربر جدید با role: "user" ساخته می‌شود
در login(): role از دیتابیس خوانده و در access token قرار می‌گیرد
در middleware: role از payload توکن استخراج و بررسی می‌شود
ساختار:

typescript

type UserRole = “user” | “employer” | “admin”

محدودیت‌های دسترسی:

/dashboard/employer/* → فقط employer یا admin
سایر مسیرها احتمالاً بر اساس role در API routes محافظت می‌شوند

⚠️ Inconsistencies و نقاط ضعف
NextAuth vs Custom JWT:

NextAuth تعریف شده اما استفاده نمی‌شود
منطق اصلی روی custom JWT است
این دوگانگی می‌تواند باعث سردرگمی شود
Middleware vs Cookie:

middleware از Authorization header می‌خواند
اگر route handlers توکن را در cookie ذخیره کنند، middleware باید تطبیق داده شود
Password Hashing:

استفاده از SHA-256 برای هش رمز عبور
⚠️ امنیتی نیست! باید از bcrypt یا argon2 استفاده شود
SHA-256 برای password hashing مناسب نیست (سریع و بدون salt)
Refresh Token Rotation:

هر بار login یک refresh token جدید ساخته می‌شود
اما در refresh endpoint مشخص نیست که آیا token rotation اتفاق می‌افتد یا خیر

۱. تکمیل Schema دیتابیس
برای یک Job Board مینیمال، این جداول کافی است:

prisma
model User {
  id            String   @id @default(uuid())
  email         String   @unique
  passwordHash  String
  name          String?
  role          Role     @default(JOBSEEKER)
  createdAt     DateTime @default(now())
  
  applications  Application[]
  refreshTokens RefreshToken[]
}

model Job {
  id          String   @id @default(uuid())
  title       String
  description String
  company     String
  location    String?
  salary      String?
  status      JobStatus @default(ACTIVE)
  createdAt   DateTime @default(now())
  
  applications Application[]
}

model Application {
  id        String   @id @default(uuid())
  userId    String
  jobId     String
  status    ApplicationStatus @default(PENDING)
  createdAt DateTime @default(now())
  
  user User @relation(fields: [userId], references: [id])
  job  Job  @relation(fields: [jobId], references: [id])
  
  @@unique([userId, jobId])
}

model RefreshToken {
  tokenId   String   @id @default(uuid())
  userId    String
  expiresAt DateTime
  revoked   Boolean  @default(false)
  
  user User @relation(fields: [userId], references: [id])
}

enum Role {
  JOBSEEKER
  EMPLOYER
  ADMIN
}

enum JobStatus {
  ACTIVE
  CLOSED
}

enum ApplicationStatus {
  PENDING
  REVIEWED
  ACCEPTED
  REJECTED
}
